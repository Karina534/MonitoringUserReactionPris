# Дизайн-документ разрабатываемого продукта

## Кейс № 1: Автоматизированная система модерации контента и анализа пользовательских реакций

### Бизнес-цель
Автоматизировать мониторинг пользовательских реакций на контент (посты, рекламу, новости) в соцсети, чтобы оперативно 
выявлять негатив, улучшать модерацию и повышать вовлеченность аудитории.

### Описание текущего процесса
Сейчас анализ тональности проводится вручную (модераторы читают выборочные комментарии) или не проводится вообще → 
медленно, дорого, субъективно. Количество комментариев, поступающих ежедневно на модерацию, в среднем доходит до 
500 000. При этом со стороны РКН ведется мониторинг качества модерации ресурса, и потенциально неприемлемые комментарии 
могут навлечь серьезные штрафы и санкции.

## 2. Методология Машинного Обучения и Данные

### 2.1 Подход к моделированию
Задача автоматического мониторинга декомпозирована на фреймворк из трех параллельных подзадач обработки естественного языка (Multi-task NLP):
1.  **Бинарная классификация токсичности:** Определение степени агрессии/опасности текста.
2.  **Мультиклассовая классификация категорий риска:** Классификация нарушений под требования регулятора (мат, разжигание ненависти, спам, призывы к противоправным действиям).
3.  **Анализ тональности (Sentiment):** Классификация на позитивный, нейтральный и негативный контент для продуктовой аналитики.

В качестве базового решения (MVP) выбрана гибридная схема: детекция явных нарушений через быстрые алгоритмы поиска по словарям стоп-слов, с последующим глубоким анализом текста легковесной квантованной языковой моделью **DistilRuBERT** в среде **ONNX Runtime / TensorRT**.

### 2.2 Метрики качества машинного обучения
* **Recall (Полнота) по классам риска — Оптимизационная метрика:** Целевое значение на этапе MVP > 96%. Минимизирует ложноотрицательные (FN) пропуски, защищая компанию от штрафов РКН.
* **Precision (Точность) — Контролируемая метрика:** Целевое значение > 80%. Защищает систему от ложных блокировок (FP) пользователей и снижает нагрузку на команду ручной модерации.

### 2.3 Пайплайн обработки данных
Спроектирован потоковый (Streaming) пайплайн данных: извлечение сырого текста из брокера сообщений -> очистка и деобфускация (борьба с маскировкой мата символами) -> subword-токенизация -> вычисление эмбеддингов нейросетью -> расчет скоров вероятности нарушения.

## 3. Системная архитектура и структура данных (Зона Архитектора / Архитектора данных)

### 3.1 Архитектурный подход
Для обеспечения изоляции основного бэкенда социальной сети и защиты от высоких нагрузок спроектирована **микросервисная асинхронная архитектура**. Публикация комментария пользователем не зависит от скорости работы нейросети — событие о новом контенте передается через распределенный брокер сообщений.

### 3.2 Обоснование распределенного характера хранения данных
Система оперирует разнородными данными с отличающимися требованиями к транзакционности, скорости чтения/записи и объемам. Спроектировано разделение по специализированным хранилищам:
* **PostgreSQL (Реляционная БД):** Хранение основных бизнес-сущностей со строгой структурой (пользователи, комментарии, вердикты моделей, кейсы ручной модерации). Гарантирует ACID-согласованность данных.
* **ClickHouse (Колоночная БД):** Хранение логов инференса нейросети (`prediction_logs`) и технических метрик. Оптимизирована для сверхбыстрой записи потока логов и выполнения агрегационных аналитических запросов.
* **Redis (In-Memory Key-Value):** Кэширование быстрых статусов проверки комментариев для снижения нагрузки на чтение из основной БД.
* **S3 (Объектное хранилище):** Хранение весов ML-моделей, артефактов обучения и выгружаемых исторических датасетов.

### 3.3 Нагрузочные характеристики и SLA
* **Нагрузка:** Средний объем — 500k комментариев/день (~5.8 RPS). Архитектура рассчитана на удержание пиковых нагрузок до **100 RPS**.
* **SLA системы:** 99.5%.
* **Задержка (Latency) ML-инференса:** До 500 мс (целевой таргет модели: 30–50 мс).
* **Скорость доставки в модерацию при критическом риске:** До 5 секунд.
